# 地图分享场景「合并显示」需求理解与初步计划

## 我的理解

我理解你的意思是：

当前项目的 map 功能里，用户可以通过分享链接导入好友的只读地图数据。你现在希望在这种“两人对照查看”的场景下，新增一种地图底色高亮模式，名字可以继续叫“合并显示”。

这个模式的目标不是把两个人的颜色直接合并成一个统一颜色，而是做出三种可视化状态，并且要同时覆盖“面高亮”和“点标记”两层：

- 两人都去过的区域：显示为一种新的“重合颜色”
- 仅当前用户去过的区域：保留当前用户自己的地图底色颜色
- 仅好友去过的区域：保留好友自己的地图底色颜色
- 两人都去过的城市点：显示为一种新的“重合点位颜色”
- 仅某一方去过的城市点：保留该用户自己的圆点颜色

这样用户在看地图时，就能一眼分辨：

- 哪些区域是两个人共同去过的
- 哪些区域只属于自己
- 哪些区域只属于好友
- 哪些城市点是两个人都去过的
- 哪些城市点只属于某一方

## 我对判定逻辑的理解

这次需求不能只看“中国地图省份底色”，还要拆成两层判断：

- 区域层：决定地图底色
- 城市点层：决定圆点颜色

并且需要按地图模式分别处理：

- 中国模式：区域层以“省份”为单位，点层以“城市点”为单位
- 世界模式：区域层以“中国以外国家”为单位，点层以“国家内城市点”为单位

也就是说，更符合这个需求的判断方式应当是：

1. 先分别统计两个人去过的中国省份集合
2. 先分别统计两个人去过的中国城市点集合
3. 再分别统计两个人去过的海外国家集合
4. 再分别统计两个人去过的海外城市点集合
5. 分别计算各集合的交集与差集
6. 地图渲染时按“区域层”和“点层”分别决定颜色

中国模式下，可视化结果应当是：

- 交集省份：`overlapColor`
- 仅自己去过：`self.regionColor`
- 仅好友去过：`friend.regionColor`
- 都没去过：保持默认底图样式
- 两人都去过的城市点：`mergePointColor` 或沿用 `mergeOverlapColor`
- 仅自己去过的城市点：`self.color`
- 仅好友去过的城市点：`friend.color`

世界模式下，可视化结果应当是：

- 两人都去过的海外国家：`overlapColor`
- 仅自己去过的海外国家：`self.regionColor`
- 仅好友去过的海外国家：`friend.regionColor`
- 国家内部如果是不同城市点：圆点继续保留各自用户颜色
- 如果两人都去过同一个海外城市点：该城市点显示为 `mergePointColor` 或沿用 `mergeOverlapColor`

这里要特别区分“区域重合”和“点位重合”：

- 如果两人在同一省不同城市都去过：
  - 省份底色应显示为重合颜色
  - 具体城市点仍保留各自颜色
- 如果两人在同一个城市都去过：
  - 该城市点应显示为重合点位颜色
  - 所在省份当然也属于重合省份
- 世界模式同理：
  - 同一国家不同城市，国家底色重合，城市点各自保留颜色
  - 同一国家同一城市，国家底色重合，城市点也显示重合颜色

## 结合当前项目代码后的现状判断

我看了当前的地图实现，项目其实已经有一些相近基础能力：

- 已有 `share` / `collaborator` 机制，支持通过分享链接导入好友地图
- 已有 `displayMode = merge`
- 已有点位重合判断逻辑，主要基于城市名、地点名、经纬度
- 已有 `mergeOverlapColor`
- 已有中国地图省份底色高亮逻辑，当前是按“某个点属于哪个用户”来给省份上色

但当前实现和这次需求之间还有几个关键差异：

- 现状更偏“点位重合染色”
- 中国区域层还没有完整支持“省份三态显示”
- 世界区域层也需要支持“国家三态显示”
- 点位层需要和区域层同时成立，但互不冲突

也就是说，这次更像是在现有 merge 模式上，同时补齐两类能力：

- 面数据渲染规则升级
- 点数据重合规则升级

## 我建议的实现方向

### 方案目标

在“已导入好友协作地图”的前提下，为双人查看场景新增一套完整的“区域 + 城市点”合并显示规则。

### 规则建议

- 默认模式：保持现在各用户各自颜色显示逻辑
- 合并显示模式：
  - 只针对已选中的两位用户生效
  - 中国模式下，省份底色采用三态颜色
  - 世界模式下，海外国家底色采用三态颜色
  - 城市 marker 点也按“重合 / 各自独有”显示颜色
  - 区域层和点层并行存在，不互相覆盖语义

### 数据处理建议

建议拆成两套聚合结果。

区域层建议形如：

```js
{
  china: {
    overlapProvinceSet,
    selfOnlyProvinceSet,
    friendOnlyProvinceSet,
  },
  world: {
    overlapCountrySet,
    selfOnlyCountrySet,
    friendOnlyCountrySet,
  }
}
```

点位层建议形如：

```js
{
  overlapLocationKeySet,
  userAOnlyLocationKeySet,
  userBOnlyLocationKeySet,
}
```

或者更通用一点：

```js
Map<areaOrLocationKey, 'overlap' | 'userA' | 'userB'>
```

这样可以分别在：

- `chinaProvinceStyle` 中按省份名决定颜色
- `worldCountryStyle` 中按国家名或国家代码决定颜色
- `CircleMarker` 中按城市点 key 决定颜色

## 初步开发计划

### 1. 明确 merge 模式的适用范围

- 确认该模式是否只在“恰好两人”时启用
- 如果当前 UI 允许选择多个用户，建议这次先收敛到“两人比较”场景
- 若选择超过两人，可先不启用省份三态，或继续沿用旧 merge 逻辑

我个人建议第一版先严格支持两人，这样产品语义最清楚，也最符合你现在的需求描述。

### 2. 追加双人区域集合计算

在当前 `allPoints` / `visible users` / `mergePairUserIds` 的基础上，新增：

- 用户 A 去过的中国省份集合
- 用户 B 去过的中国省份集合
- 中国省份交集集合
- A 独有中国省份集合
- B 独有中国省份集合
- 用户 A 去过的海外国家集合
- 用户 B 去过的海外国家集合
- 海外国家交集集合
- A 独有海外国家集合
- B 独有海外国家集合

这里建议：

- 中国区域层只统计 `point.regionMeta.isChina === true` 且 `provinceName` 有效的数据
- 世界区域层只统计 `point.regionMeta.isChina !== true` 且 `countryCode / countryName` 有效的数据

### 3. 追加双人城市点集合计算

除了区域层，还需要单独算城市点是否重合。

建议沿用或扩展现有 `pointLocationKeys` 思路，但语义要更明确：

- 中国模式：重点使用 `country + province + city`
- 世界模式：重点使用 `country + city`
- 若城市名缺失，再视情况回退到 `place` 或经纬度近似 key

最终得到：

- 两人都去过的城市点 key 集合
- A 独有城市点 key 集合
- B 独有城市点 key 集合

这样 marker 渲染时就能正确区分：

- 同区域不同城市：区域合并，点不合并
- 同区域同城市：区域合并，点也合并

### 4. 改造中国与世界地图底色渲染逻辑

当前项目已有 `chinaProvinceStyle` 和 `worldCountryStyle`。这一层可以改成：

- 若当前不在合并显示模式：沿用原逻辑
- 若当前在双人合并显示模式：
  - 中国地图：
    - 交集省份使用 `mergeOverlapColor`
    - A 独有省份使用 A 的 `regionColor`
    - B 独有省份使用 B 的 `regionColor`
  - 世界地图：
    - 交集国家使用 `mergeOverlapColor`
    - A 独有国家使用 A 的 `regionColor`
    - B 独有国家使用 B 的 `regionColor`
  - 其他区域使用默认未点亮样式

这样对现有结构侵入最小。

### 5. 改造城市点颜色渲染逻辑

当前项目已有 marker / `CircleMarker` 颜色逻辑，这一层建议补成：

- 若当前不在合并显示模式：沿用原逻辑
- 若当前在双人合并显示模式：
  - 两人都去过的城市点：使用 `mergePointColor`
  - A 独有城市点：使用 A 的 `color`
  - B 独有城市点：使用 B 的 `color`

这里有一个实现选择：

- 方案 A：城市点重合直接沿用现有 `mergeOverlapColor`
- 方案 B：单独新增 `mergePointColor`

我个人更建议先用方案 A，这样配置成本更低；如果后面觉得“区域色”和“点位色”需要视觉区分，再扩展为两个颜色配置。

### 6. 调整文案与图例

为了让用户能看懂，建议在 merge 模式下增加清晰图例：

- 区域重合：某颜色
- 我的区域：我的区域颜色
- 好友区域：好友区域颜色
- 城市重合：某颜色
- 我的城市点：我的点位颜色
- 好友城市点：好友点位颜色

如果当前顶部已有 merge 颜色配置，可以继续保留；但建议文案从“同城重合”扩展成更完整的“共同去过区域 / 城市”的表述，避免用户误解。

### 7. 验证边界情况

需要重点验证：

- 好友没有中国点位时，是否还能进入该模式
- 两人都没有省份信息但有经纬度时，底色是否应完全不变
- 同省不同城市是否按“重合省份”显示
- 同国不同城市是否按“重合国家”显示
- 同省不同城市时，城市点是否仍分别显示各自颜色
- 同国同城市时，城市点是否正确显示为重合颜色
- 导入多个好友时，当前比较对象是谁
- 用户隐藏某个协作好友后，是否自动退出该模式
- 海外国家名与国家代码不一致时，是否能正确归并
- 中国数据是否严格排除在世界国家重合统计之外

## 我认为最需要先确认的一个产品点

现在最值得先定死的口径有两个：

**1. 区域层重合按什么算？**

- 中国模式按省份
- 世界模式按国家

**2. 点位层重合按什么算？**

- 中国模式按“省份 + 城市”
- 世界模式按“国家 + 城市”
- 缺失城市名时，回退到 `place` 或经纬度近似匹配

按你这次补充后的描述，我更倾向于理解为：

- 地图底色层：按“省份重合”判断
- 世界底色层：按“国家重合”判断
- marker 点层：按“城市重合”判断
- 区域重合和点位重合是两套并行规则，不互相替代

这样既不浪费当前已有点位 merge 逻辑，也能满足你想要的“中国省份 / 世界国家 / 城市点”三层语义。

## 我建议的第一版落地口径

如果让我直接开始做，我会按下面这个第一版来实现：

1. 仅在中国地图下生效
2. 仅支持两人比较
3. 中国模式下，省份底色按交集 / 各自独有三种状态显示
4. 世界模式下，国家底色按交集 / 各自独有三种状态显示
5. 城市 marker 点按“同城市重合 / 各自独有”显示
6. 中国区域重合按“省份”，世界区域重合按“国家”
7. 点位重合按“城市”，缺失城市名时再回退其他 key

上面第 1 条我这里修正一下。按照你刚刚的补充，第一版不应该只在中国地图下生效，而应该中国和世界模式都支持，只是区域层级不同。
