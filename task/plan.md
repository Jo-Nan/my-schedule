# Day 项目升级实施计划

## 1. 目标概述

本次升级目标不是重做现有界面，而是在尽量保持当前使用体验、Logo 和主体布局不大改的前提下，把项目从“单用户 + 前端本地鉴权 + 单份计划数据”升级为“多用户 + 服务端认证 + 用户数据隔离 + 管理员后台 + 邮件通知 + 定时邮件简报”的版本。

同时补充两个细节要求：

- 天气区域不再显示 `Fetching weather...` 之类的加载文案；有天气就显示，没有就不显示。
- 每天北京时间 `08:05` 给每个用户发送“今日计划简报”。
- 每天北京时间 `22:05` 给每个用户发送“当日完成情况进度 + 第二天计划表”。

---

## 2. 当前现状与限制

### 当前系统现状

- 登录方式仍是前端密码闸门，没有邮箱注册、用户体系和服务端会话。
- 当前计划数据只有一份共享数据，所有数据默认属于同一个人。
- 同步接口只围绕一份 `plans.json` 工作，不支持按用户隔离。
- 备份逻辑也是对单份共享数据进行备份。
- 如果继续使用纯 GitHub Pages 静态托管，则无法可靠支持注册、登录、邮件发送、管理员管理和定时任务。

### 结论

为了实现这次需求，项目需要从“纯静态前端模式”升级为“前端 + 服务端 API + 数据库 + 定时任务”的架构。

---

## 3. 推荐架构

### 前端

- 保留当前 React + Vite 项目。
- 尽量不改 Logo，不大改普通用户主界面结构。
- 登录层从“单密码输入”改成“登录 / 注册”两种入口。

### 后端

- 继续使用同仓库下的服务端 API 路由。
- 推荐部署到 `Vercel`，便于承载 API、邮件发送和 Cron 定时任务。

### 数据层

- 推荐使用 `Postgres`。
- 所有核心数据都按 `userId` 做强关联。
- 邮箱是登录标识，但系统内部仍以 `userId` 作为主关联键，避免以后用户改邮箱时导致数据关系错乱。

### 邮件能力

- 推荐使用 `Resend` 或标准 SMTP 服务。
- 邮件能力用于：
  - 用户给管理员发消息通知。
  - 每日 `08:05` 今日计划简报。
  - 每日 `22:05` 当日完成进度 + 次日计划表。

### 定时任务

- 推荐使用 `Vercel Cron`。
- 其 Cron 表达式使用 UTC 时区，因此需要换算北京时间：
  - 北京时间 `08:05` = UTC `00:05`，Cron：`5 0 * * *`
  - 北京时间 `22:05` = UTC `14:05`，Cron：`5 14 * * *`

## 4. 核心业务规则

### 4.1 注册与登录

- 新增邮箱注册功能。
- 注册字段：
  - 邮箱（必填，唯一）
  - 密码（必填）
  - 用户名（可选，默认空字符串）
- 登录支持使用邮箱 + 密码。
- 密码必须只保存哈希值，不能在前端或数据库中存明文。
- 登录成功后由服务端下发安全会话，推荐 `HttpOnly Cookie`。

### 4.2 超级管理员

- 超级管理员账号信息如下：
  - 用户名：`NanMuZ`
  - 邮箱：`nanqiao.ai@gmail.com`
  - 密码：`571428`
- 该账号需要通过服务端初始化写入，不能继续像现在这样把鉴权逻辑硬编码在前端。
- 当前现有的所有计划数据，默认迁移并归属到超级管理员名下。

### 4.3 普通用户数据隔离

- 普通用户登录后，看到的界面功能与当前版本大体保持一致。
- 但所有计划数据、同步数据、备份数据都只属于当前用户本人。
- 用户之间互相不可见，不可互相修改。

### 4.4 管理员权限

- 超级管理员除了管理自己的数据，还能管理其他所有用户。
- 管理员能力至少包括：
  - 查看用户列表
  - 新增用户
  - 删除用户
  - 查看指定用户数据
  - 恢复指定用户某一天的数据
  - 查看用户给管理员发送的消息记录
- 超级管理员界面与普通用户界面不同，需要独立的管理入口或管理面板。

### 4.5 用户消息到管理员邮箱

- 每个用户都可以给管理员发送消息。
- 用户发送消息后，需要同时完成两件事：
  - 消息落库保存
  - 给管理员邮箱发送通知邮件
- 管理员收到的邮件中必须包含：
  - 用户邮箱
  - 用户名
  - 具体消息内容
  - 发送时间

### 4.6 每日计划邮件

#### 每日早报

- 每天北京时间 `08:05`，给每个用户发送一封“今日计划简报”邮件。
- 邮件内容至少包括：
  - 用户当天的计划列表
  - 计划数量
  - 完成状态摘要（如果有）

#### 每日晚报

- 每天北京时间 `22:05`，给每个用户发送一封“当日完成情况进度 + 第二天计划表”邮件。
- 邮件内容至少包括：
  - 当日已完成 / 进行中 / 未完成统计
  - 当日计划明细摘要
  - 第二天计划表

## 5. 数据模型设计

建议新增如下核心数据表。

### 5.1 `users`

用于保存账号信息。

建议字段：

- `id`
- `email`（唯一）
- `username`（可空，默认空字符串）
- `password_hash`
- `role`（`admin` / `user`）
- `is_active`
- `created_at`
- `updated_at`

### 5.2 `plans`

用于保存用户当前有效计划。

建议字段：

- `id`
- `user_id`
- `event`
- `date`
- `time`
- `person`
- `ddl`
- `progress`
- `status`
- `updated_at`
- `created_at`

### 5.3 `plan_snapshots`

用于保存按用户归档的每日快照，支撑“恢复某一天数据”。

建议字段：

- `id`
- `user_id`
- `snapshot_date`
- `plans_json`
- `source`（如 `auto_backup` / `manual_restore_point`）
- `created_at`

### 5.4 `messages`

用于保存用户给管理员发送的消息。

建议字段：

- `id`
- `user_id`
- `user_email`
- `username`
- `content`
- `created_at`

### 5.5 `daily_digest_logs`

用于记录邮件发送情况，避免重复发送，方便排错。

建议字段：

- `id`
- `user_id`
- `digest_type`（`morning` / `night`）
- `digest_date`
- `send_status`
- `error_message`
- `created_at`

## 6. 接口与模块改造计划

### 6.1 认证模块改造

当前前端 `AuthOverlay` 需要升级为服务端认证方案。

需要新增：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

注册成功后：

- 新建用户记录
- 自动建立该用户的数据空间
- 返回登录态或引导到登录态

### 6.2 计划数据接口改造

当前 `load-plans` / `save-plans` 是单用户共享数据接口，需要改为基于当前登录用户的接口。

建议拆分为：

- `GET /api/plans`
- `POST /api/plans`
- `PATCH /api/plans/:id`
- `DELETE /api/plans/:id`

管理员额外接口：

- `GET /api/admin/users`
- `POST /api/admin/users`
- `DELETE /api/admin/users/:id`
- `GET /api/admin/users/:id/plans`
- `GET /api/admin/users/:id/snapshots`
- `POST /api/admin/users/:id/restore`

### 6.3 消息接口

新增：

- `POST /api/messages`

职责：

- 校验用户身份
- 保存消息记录
- 给管理员邮箱发送通知邮件

### 6.4 定时任务接口

新增：

- `GET /api/cron/send-morning-digest`
- `GET /api/cron/send-night-digest`

职责：

- 遍历有效用户
- 生成每个用户对应邮件内容
- 发送邮件
- 写入发送日志

## 7. 前端改造范围

### 7.1 保持不大改的部分

- Logo 不改。
- 普通用户主界面布局和交互保持当前整体风格。
- 日 / 周 / 月 / 年视图的大方向保持不变。

### 7.2 必须改造的部分

- 登录页改为“登录 / 注册”双入口。
- 增加用户会话状态获取逻辑。
- 本地 `localStorage` 不能再作为最终真实数据源，只能做轻缓存或不用。
- 当前计划读写改为请求服务端。
- 增加“给管理员发消息”的入口。

### 7.3 管理员界面

管理员登录后进入不同界面或多一个管理面板。

至少包含：

- 用户列表
- 用户创建 / 删除
- 用户计划查看
- 某日数据恢复
- 消息查看

### 7.4 天气显示调整

- 移除 `Fetching weather...` 加载提示文案。
- 当天气数据存在时显示天气。
- 当天气接口失败、为空或尚未返回时，不显示天气区域占位文案。

## 8. 数据迁移与备份策略

### 8.1 旧数据迁移

- 当前已有的所有计划数据全部迁移到超级管理员账号名下。
- 迁移后普通用户从空数据开始。

### 8.2 备份策略升级

- 原先的单份共享备份，升级为按用户生成快照。
- 每天至少为每个用户保留一个可恢复快照。
- 管理员可以在后台恢复指定用户某一天的快照。

### 8.3 恢复原则

- 恢复操作必须按用户执行，不能全局覆盖所有用户数据。
- 恢复前建议额外创建一个恢复点，避免误操作后无法回退。

### 第一阶段：后端基础与认证

目标：先把“多用户”最底层基础打好。

3. 把当前旧数据迁移到超级管理员账号。

1. 新增管理员专用界面。

5. 实现恢复某用户某日数据。

### 第四阶段：消息与邮件通知

目标：打通用户与管理员的消息链路。

任务：

1. 新增用户发消息入口。
2. 新增消息保存接口。

### 第五阶段：定时邮件简报

任务：

1. 实现今日计划简报生成逻辑。

任务：

1. 去掉天气加载文字占位。

## 11. 风险与注意事项

- 超级管理员密码不能继续写在前端代码里，必须迁移到服务端安全存储。
- 如果继续使用 GitHub Pages 纯静态部署，以上需求无法完整实现。
- 邮件发送需要配置独立邮件服务及发信域名/账号。
- 定时任务平台一般默认使用 UTC，需要明确按北京时间换算。
- 删除用户和恢复数据都属于高风险操作，后台需要增加二次确认。

## 12. 下一步建议

建议按以下优先级开始落地：

1. 先完成数据库、认证和超级管理员初始化。
2. 再完成“每个用户只能看到自己的计划数据”。
3. 然后补管理员后台、消息和邮件。
4. 最后完成 `08:05` 与 `22:05` 两个自动邮件任务，以及天气显示细节收尾。

如果开始正式开发，第一批最值得先交付的功能是：

- 邮箱注册 / 登录
- 超级管理员账号接入
- 当前旧数据迁移到管理员名下
- 普通用户计划数据隔离
